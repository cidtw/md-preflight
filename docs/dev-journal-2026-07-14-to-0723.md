#### 1.1 배경 분석 (대화·티켓 요약)

**서비스 재정의 (대전제)**  
- 대상: 유통·물류 프로모션/머천다이징 담당 MD·개인사업자  
- 가치: 단순 반복이지만 실패 시 고객경험·매출에 직결되는 **행사 사전 검수**  
- 경쟁 맥락: ERP가 방대하더라도 말단 영업관리자의 디테일 통제가 성과 키 → Preflight는 “치트키” 포지션  
- final vision: 고정 3파일 alpha를 넘어, 비정형·단일 chunk도 frame 분리 후 검수하고 **수신 채널로 retrieve** 하는 자동화 인터페이스  

**중간발표 Q&A에서 확정된 사실**  
1. 현재 입력은 고정 스프레드시트(csv/xlsx)뿐 → final에서 확장 약속  
2. 타겟 유저는 영업관리 담당·개인사업자, 교육 목표 **1시간 이내**  
3. **column key 변경 시나리오는 테스트하지 않음** → final에서 별칭·유사 키 수렴 방향  

**자가진단**  
1. 구조는 납득되나 실무 competitive edge 스토리가 약함  
2. Vercel SPA 일체형 + Clerk 외부 인증이 테스트·UX에 불편 → 화면 split + (장기) 자체 멤버 설정  

#### 1.2 의사결정

| 결정 | 내용 | 이유 |
|------|------|------|
| D-A | 7/24까지 **중간 릴리스** (“유연 입력 + 분리 UX”), final RC 전 범위는 백로그 | 10일 창에서 chunk 분해·채널 자동화·자체 DB 교체는 리스크 과다 |
| D-B | **T48 별칭을 최우선** | Q3 미검증이 기술·데모 모두 최대 구멍; 룰 10개를 건드리지 않고 ingest만 확장 가능 |
| D-C | **T49 해시 라우트 4화면** (풀 MPA/Next 분리 아님) | FastAPI static SPA 유지 비용 최소 · 발표용 “화면 분리” 스토리 즉시 성립 |
| D-D | 별칭은 **명시 테이블**(리뷰 가능), fuzzy ML 금지 | 결정론·감사 가능 원칙과 정합 |
| D-E | 설정 화면은 **읽기 전용 카탈로그**부터 | 1시간 교육 스토리 지원; 사용자 정의 매핑 UI는 후속 |

## 4. 부록 — 7/14 분석 메모 (대화 원문 압축)

### 4.1 서비스가 무엇인가
물류·유통 프로모션 실행 직전, 놓치면 고객경험·매출을 해치는 반복 검수를 **결정론 규칙**으로 자동화하는 도구. LLM은 설명·체크리스트만.

### 4.2 무엇이 불편했는가
프랜차이즈 전국 전개 시 프로모션 실행 품질 편차 · 비협조 점포 · 본사 기획 대비 기댓값 미달. ERP로 “불가능”한 문제는 아니나, **말단 담당자 디테일**이 성과 분기점.

### 4.3 최종 인터페이스 비전 (alpha → RC)
| alpha | final RC (장기) |
|-------|-----------------|
| 프로모션/마스터/재고 고정 3파일 | 비정형·단일 chunk도 frame 분리 검수 |
| 고정 컬럼명 | 상황별 별칭 수렴 · 경고 분류 |
| 업로드 UI 중심 | 수신 채널로 결과 retrieve 자동화 |

### 4.4 7/14에 선택한 “수정 가능한 빌드”의 의미
- 별칭 표 = **파일 1곳 패치로 ERP 고객 확장**  
- 해시 라우트 = **빌드 스텝 없이** 화면 정보 구조 실험  
- 설정 카탈로그 = 교육·발표·온보딩 카피의 단일 출처 후보  

이 세 가지는 final RC의 거대 기능을 막지 않으면서, 중간발표 약점을 **즉시 데모 가능한 형태**로 줄인다.
