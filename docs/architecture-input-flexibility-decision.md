# 의사결정: 입력 유연성 — “3프레임에 맞출 것인가, 구조를 버릴 것인가”

**작성**: 2026-07-14  
**맥락**: 중간발표 핵심 피드백 — *어떤 종류·형태의 프로모션 입력이 와도 정제(검수) 가능해야 한다.*  
**현상**: 제품은 여전히 **프로모션 계획 / 상품 마스터 / 재고** 3슬롯 + 정규 컬럼 집합에 묶여 있다. (T48 별칭은 *헤더 이름*만 완화.)

---

## 1. 문제를 두 층으로 분리

피드백 문장을 한 덩어리로 받으면 설계가 흔들린다. **입력 표면**과 **판정 모델**을 나눠야 한다.

| 층 | 질문 | 지금 | 피드백이 요구하는 것 |
|----|------|------|----------------------|
| **A. 입력 표면** | 무엇을 올릴 수 있나? | 파일 3개, 역할 고정, csv/xlsx | 양식·개수·청크·프로모션 종류가 달라도 *들어와야* 함 |
| **B. 판정 모델** | 무엇을 보고 이슈를 내나? | `PreflightContext` = 3 테이블 조인 + 10 룰 | 같은 리스크 논리(가격·기간·마진·재고·증정)를 **재현·감사 가능**하게 |

**핵심 오해**  
“3프레임워크에 묶였다” = (1) 업로드 UI가 3칸이라서가 아니라,  
(2) **룰이 읽는 언어가 3개 정규 프레임**이기 때문이다.  
(1)은 바꿀 수 있고, (2)를 없애면 D3(결정론 판정)와 10룰 자산이 같이 무너진다.

---

## 2. 선택지 비교

### 옵션 X — 들어오는 데이터를 **정규 프레임으로 정제(어댑트)** 한다 ★ 권고

```
[임의 입력]
  단일 xlsx 여러 시트 · 파일 N개 · 한글 ERP 덤프 · (나중) 메일 첨부
        │
        ▼
  Adapter / Mapper  (역할 추론 · 헤더 별칭 · 시트 분리 · 매핑 확정)
        │
        ▼
  Canonical frames  (promotion · product · inventory)  ← 판정 언어 유지
        │
        ▼
  기존 Rule Engine (10룰, 확장 가능)
```

| 장점 | 단점 |
|------|------|
| **D3 유지** — 판정 계약·테스트·rule_set_version 그대로 | “완전 자유 스키마”는 아님 |
| 이미 만든 10룰·체크리스트·조인 불변식 **재사용** | 매퍼 품질이 제품 품질이 됨 |
| 중간발표 약속(별칭·비정형 청크 분리)과 **같은 방향** | 매핑 UI/확정 단계 필요 |
| 경쟁 스토리: “ERP마다 달라도 **같은 검수 언어**로 정제” | 프로모션 *유형*이 늘면 룰 팩 추가 필요 |

### 옵션 Y — **구조 자체를 바꾼다** (프레임 폐기 · 범용 스키마/LLM 분류)

```
[임의 입력] → (LLM 또는 완전 동적 스키마) → 자유 이슈
```

| 장점 | 단점 |
|------|------|
| 마케팅 문장상 “뭐든 넣어도 됨”에 가깝게 들림 | **판정 비결정론 / 감사 불가** — 프로젝트 정체성 붕괴 |
| 짧은 데모용 와우 가능 | 10룰·T8 계약 테스트·포트폴리오 스토리 전면 재작성 |
| | 7/24 창에서 완료 불가, 회귀 폭발 |

→ **Y는 final RC 전체 재설계 에픽이지, 지금 고를 기본 전략이 아님.**  
(단, “프로모션 유형별 룰 팩” 정도의 *확장*은 X 위에서 가능.)

### 옵션 Z — 하이브리드 (권고안을 단계로 쪼갠 것)

| 단계 | 내용 | 판정 모델 |
|------|------|-----------|
| **Z0 (완료)** | 컬럼 별칭 T48–T53 | 3프레임 고정 |
| **Z1** | 역할 매핑 UI: 업로드 N파일/시트를 promo/master/inv에 **할당** | 동일 |
| **Z2** | 단일 워크북 자동 분리(시트명·헤더 시그니처) + 사용자 확정 | 동일 |
| **Z3** | 프로모션 **유형 프로필** (할인 / 1+1 / 증정…) → 룰 서브셋·추가 필드 | 프레임 확장(필드 추가), 폐기는 아님 |
| **Z4** | 채널 retrieve · 완전 비정형 chunk | Z2+ 전제 |

Z = **X를 시간축으로 깐 것**. Y로 점프하지 않는다.

## 3. 권고 결정 (제안)

### 결정문 (안)

> **내부 판정 모델(정규 3프레임 + 조인 + 결정론 룰)은 유지한다.**  
> **입력 표면은 “무엇이든 3프레임으로 정제하는 어댑터”로 연다.**  
> 구조 폐기(옵션 Y)는 하지 않는다. 대신 어댑터·매핑·유형 프로필로 피드백을 충족한다.

### 왜 이 결정인가

1. **피드백의 본심**은 “파일 3개 슬롯”이 아니라 “**양식이 달라도 검수가 된다**”.  
   별칭(T48)은 헤더 층만 열었고, **역할·시트·청크·프로모션 종류**는 아직 닫혀 있다 → 다음 공은 어댑터.
2. **D3가 제품의 competitive edge**다. 구조를 버리면 “AI가 알아서 검수” 범용 툴과 차별이 사라진다.
3. 티켓 원문의 final RC 문장 자체가  
   *“하나의 chunk → 여러 frame으로 분리 → 오류 분류”*  
   즉 **프레임으로 정제**를 전제한다. 프레임 폐기가 아님.
4. 구현 리스크: 어댑터는 **ingest 앞단 추가**로 끝난다. 룰 전면 재작성보다 안전.

### “3프레임이 모든 프로모션을 담나?”에 대한 답

- **담는 것**: 가격·기간·마진·SKU 정합·재고·기본 증정 조건 — 유통 사전등록 리스크의 공통 분모.  
- **안 담는 것(아직)**: 점포별 차등, 카드사 제휴, 시간대 특가 등 → **유형 프로필(Z3)** 로 필드/룰을 *추가*하지, 공통 프레임을 버리지 않음.  
- 프레임 이름은 UI에서 “프로모션 / 마스터 / 재고” 대신  
  **“행사 라인 · 상품 기준가 · 공급(재고)”** 처럼 역할 언어로 풀어 설명할 수 있다.

## 4. 지금 구조가 묶인 지점 (구현 맵)

| 지점 | 고정 내용 |
|------|-----------|
| API | `promotion_plan` · `product_master` · `inventory` 3 multipart 필드 |
| UI | `FIELDS` 3 dropzone |
| domain | `SourceFile` enum 3 · `ALL_SOURCE_COLUMNS` |
| ingest | 3 DF → join on `product_code` |
| rules | `PreflightContext` 3+joined 전제 |
| 완화 완료 | 헤더 별칭, missing 힌트, catalog |

**바꾸면 안 되는 것(당분간)**  
`PreflightContext` + 룰 입력 계약 + D3 테스트.

**바꿔야 하는 것(다음)**  
업로드 계약·역할 매핑·시트 분리 — 전부 **Context 생성 이전**.
