# 방향성 — 매장 특화 ROP 재조정

## 문제

프로모션 검수처럼 정량 파일이 많은 검토가 아니라, **운영 매장 특성**과 **품목 재고 운영 기준(Lead Time · ROP)** 을 맞추는 의사결정이 필요하다.

- 업계/사내 표준 LT·ROP는 채널 평균에 가깝다.
- 실제 매장은 규모·객단가·상권·접근성에 따라 물류 지연과 수요 변동이 다르다.
- 담당자는 “왜 이 값인가”에 대한 **근거**가 있어야 발주 정책을 바꿀 수 있다.

## 서비스 한 줄

매장 파라미터 + 품목·소진량을 입력하면, 사전 가중치 테이블과 지식 베이스 매칭으로  
**LT는 입력 고정**, 조정 레버(**안전재고 · 1회 발주량 · 발주 주기 · ROP**)와 근거 리포트를 반환한다.

## 핵심 축

| 축 | 내용 |
|----|------|
| 물류 수용 | 연면적 CAPA 점수, 접근성 → 물류 리스크 버퍼(개) |
| 수요 변동 | 상권 유형, 객단가 회전 가중치, KB 안전계수 Z |
| 표준 대비 | 채널/사내 baseline vs 매장 맞춤 재조정 |
| 운영 제약 | CAPA 초과 시 ROP·Q 상한 + 다회 소량 발주 제안 |

## 원칙

1. **결정론 연산** — 동일 입력 → 동일 ROP/SS/Q/근거 수치  
2. **입력 불일치 시 우선순위** — 규모·객단가 선택을 유형보다 우선  
3. **근거 필수** — 숫자만 주지 않고 물류버퍼 / 수요·운영레버 / geo / CAPA 블록 서술  
4. **예시 문서 문구 복붙 금지** — 입력 기반 동적 생성  
5. **모듈 3단** — input → analyze → output 만 확장  
6. **LT 고정** — 계약·표준 LT는 출력에서 재조정하지 않음  
7. **근거 3층 (국면 XI)** — L1 표준 이론 · L2 문헌/기관 수치 · L3 명시 engineering assumption  
   상세: [`docs/evidence/`](../evidence/) · 시연은 스코프 축소로 논리 깊이 우선
